## 2. 失败方案详细分析

### 2.1 F01：Docker Compose 环境对等方案

#### 方案概述
使用 Docker Compose 编排 8 个容器（mock-wikipedia / mock-anthropic / mock-openai / mock-gemini / local-gitea / local-minio / local-nginx / local-mailhog），实现本地与生产环境架构一致。

#### 失败原因（详细）

1. **资源占用过大**
   - Docker Desktop 在 Mac 上默认占用 4GB+ 内存
   - 8 个容器同时运行需 8GB+ 内存
   - 用户硬件未知，按 8GB Mac 评估会触发 OOM
   - 与 Hugo 构建（需 4GB 内存）+ Python 管道（需 1GB 内存）冲突

2. **性能损耗明显**
   - Docker volume 在 Mac 上有 2-3 倍 IO 性能损耗
   - Hugo 10 万页构建从 8min 涨到 20min+
   - 与性能预算（< 10min）严重冲突
   - 已知问题：macOS 文件系统与 Linux 容器文件系统不兼容导致 ENOSPC

3. **容器编排复杂度高**
   - 8 个容器需维护 8 份 Dockerfile
   - docker-compose.yml 配置复杂（网络/卷/依赖关系）
   - 容器间通信需调试，调试成本高
   - 一旦容器编排出错，整个开发环境崩溃

4. **Gitea Actions 兼容性风险**
   - Gitea Actions 是 GitHub Actions 的开源克隆，但兼容性不完整
   - 部分第三方 Action 无法使用
   - 行为差异可能导致本地通过但生产失败
   - 维护成本高于直接使用 `act` 工具

5. **与原则冲突**
   - 与"最小化原则"冲突：8 个容器 vs 1 个 Flask Mock Server
   - 与"风险最小化"冲突：容器编排引入新风险
   - 与"暴露异常"冲突：容器日志隔离，问题定位困难

#### 避坑指南

**坑 1：盲目追求"环境一致性"**
- **症状**：认为"本地与生产一致"必须用 Docker
- **真相**：本地开发用 Flask Mock + Hugo server 已足够"走通全链路"
- **正确做法**：本地用 Provider 抽象 + Flask Mock，生产用真实 API + Cloudflare Pages，环境差异通过 `.env` 切换

**坑 2：忽视 Docker volume 性能损耗**
- **症状**：在 Linux 服务器上 Docker 表现良好，移植到 Mac 后性能骤降
- **真相**：macOS 文件系统与 Linux 容器文件系统不兼容，volume 性能损耗 2-3 倍
- **正确做法**：Mac 本地开发避免使用 Docker volume 处理大量文件（如 Hugo 10 万页）

**坑 3：使用 Gitea Actions 替代 GitHub Actions**
- **症状**：本地 Gitea Actions 通过，推送到 GitHub 后失败
- **真相**：Gitea Actions 兼容性不完整，部分 Action 不支持
- **正确做法**：使用 `act` 工具本地模拟 GitHub Actions，保持 workflow 文件一致

**坑 4：容器编排出错导致全栈崩溃**
- **症状**：修改某个容器配置后，整个 Docker Compose 启动失败
- **真相**：容器间依赖关系复杂，单点变更可能级联失败
- **正确做法**：避免容器化本地开发环境，用单一 Flask Mock Server 统一响应

#### 替代方案
- 用 **Flask Mock Server**（端口 8765）替代 8 个容器
- 用 **`act` 工具** 替代 Gitea Actions
- 用 **本地 Hugo server** 替代 local-nginx
- 用 **本地 Git** 替代 local-gitea

### 2.2 F02：Gitea Actions 替代 GitHub Actions

#### 方案概述
在本地搭建 Gitea + Gitea Actions，模拟 GitHub Actions 工作流。

#### 失败原因

1. **兼容性风险**
   - Gitea Actions 仅兼容 GitHub Actions 的子集
   - 第三方 Action（如 `actions/cache@v4`、`actions/upload-artifact@v4`）可能不支持
   - 自定义 Action 的 `composite` 类型支持不完整

2. **维护成本高**
   - Gitea 升级需同步 Gitea Actions Runner
   - Runner 注册/管理复杂
   - 日志查看不如 GitHub 直观

3. **行为差异**
   - 触发器（push/pull_request/schedule）行为不完全一致
   - Secrets 管理机制不同
   - 环境变量默认值不同

#### 避坑指南

**坑 1：Gitea Actions 通过 ≠ GitHub Actions 通过**
- **症状**：本地 Gitea Actions 全绿，推送 GitHub 后失败
- **真相**：兼容性不完整，部分 API 行为差异
- **正确做法**：使用 `act`（nektos/act）本地模拟，与 GitHub Actions 共享 workflow 文件

**坑 2：Runner 注册失败**
- **症状**：Gitea Actions Runner 无法注册到 Gitea 实例
- **真相**：网络配置 / Token 过期 / Runner 标签不匹配
- **正确做法**：避免使用 Gitea Actions，用 `act` 替代

#### 替代方案
- 使用 **`act`**（https://github.com/nektos/act）本地运行 GitHub Actions
- 关键 workflow 在生产环境复验

### 2.3 F03：最小化风险防控方案

#### 方案概述
仅关注 Top 3 风险（R01 Mock 失真、R02 网络不可达、R03 Hugo OOM），轻量监控 + 口头应急预案 + 延迟风险登记。

#### 失败原因

1. **风险覆盖不足**
   - 仅覆盖 3 项极高风险，遗漏 12 项其他风险
   - 中高风险（R09 数据合规、R10 API Key 泄露）未覆盖
   - 一旦发生未识别风险，缺乏应急预案

2. **与原则冲突**
   - 与"系统稳定高于一切"原则冲突
   - 与"风险最小化"原则冲突
   - 与"证据驱动"原则冲突（口头应急预案不可追溯）

3. **技术债务隐性积累**
   - 未显式登记技术债务，债务积累不可见
   - 遇到时再登记可能导致债务集中爆发
   - 月度评审流于形式

#### 避坑指南

**坑 1：轻视风险防控导致生产事故**
- **症状**：开发阶段顺风顺水，生产迁移时集中爆发
- **真相**：未识别风险在生产环境暴露，无应急预案
- **正确做法**：完整风险矩阵（15 项）+ 8 项技术债务显式登记 + BCP/DRP

**坑 2：口头应急预案无法执行**
- **症状**：紧急时刻发现"我们之前讨论过但没记录"
- **真相**：口头讨论无文档化，无法追溯
- **正确做法**：BCP/DRP 文档化，明确触发条件 / 负责人 / RTO / 回滚方案

#### 替代方案
- 采用**分级风险防控 + 显式技术债务登记**（风险分析师方案1）
- 15 项风险矩阵 + 8 项技术债务登记 + BCP/DRP + CI 风险门禁

### 2.4 F04：同步版管道（subprocess.run 串行）

#### 方案概述
用 `subprocess.run` 串行调用 7 阶段，单日 Mock 数据 ~15min。

#### 失败原因

1. **性能差**
   - 单日 Mock 管道 15min vs asyncio 版 3min（5 倍差距）
   - 366 天全量 Mock 数据 90 小时 vs 18 小时
   - 调试反馈循环 15min/次 vs 30s/次（缓存命中）

2. **不能复现生产时序**
   - 串行调用无法暴露并发问题（如 fd 耗尽、rate limit）
   - 与生产环境（asyncio + httpx）行为不一致

3. **不能利用 diskcache**
   - 同步版无并发优势，diskcache 加速效果不明显
   - 缓存命中率统计无意义

#### 避坑指南

**坑 1：同步版通过 ≠ 生产版通过**
- **症状**：同步版测试全绿，切换 asyncio 后崩溃
- **真相**：并发问题（race condition / fd 耗尽 / rate limit）无法在同步版暴露
- **正确做法**：直接使用 asyncio 版，保留同步版仅作小数据量调试 fallback

**坑 2：调试反馈循环过长**
- **症状**：每次改 prompt 后需等 15min 才能看到结果
- **真相**：同步版无 diskcache 加速，每次重新调用 Mock API
- **正确做法**：asyncio + diskcache，二次运行 30s 内出结果

#### 替代方案
- 采用 **asyncio + Semaphore(16) + diskcache**（性能工程师方案2）
- 阶段间串行，阶段内并发（30 地区 / 10 语种）
- 保留同步版作为 fallback，1 个月后废弃

### 2.5 P01：VCR.py 生产数据回放方案（延后）

#### 方案概述
首次运行时用 VCR.py 录制真实 API 响应，后续运行直接回放 cassette，Mock 数据 100% 真实。

#### 延后原因

1. **首次录制需网络访问**
   - 用户当前无国外网络访问，无法录制真实响应
   - 强行录制会卡死在第一次调用

2. **cassette 维护成本高**
   - API 升级需重新录制
   - cassette 文件体积大（每个 cassette 1-10MB）
   - 多版本管理复杂

3. **当前阶段非必需**
   - 人工编写 fixture 已足够覆盖 happy path
   - Mock 失真风险通过契约测试 + Phase 4 真实样本对比已可控

#### 启用时机
Phase 4（生产迁移阶段）：
1. 用户获得网络访问后，用 VCR.py 录制真实响应
2. 对比 Mock 差异，识别失真点
3. 更新 Mock fixture，提升仿真度

#### 避坑指南

**坑 1：强行启用 VCR.py 导致卡死**
- **症状**：VCR.py 首次运行时卡在网络请求，无法继续
- **真相**：VCR.py 录制模式需真实网络访问，无网络时无法工作
- **正确做法**：当前阶段用人工编写 fixture，Phase 4 再启用 VCR.py

**坑 2：cassette 与 API 版本不匹配**
- **症状**：回放 cassette 时字段缺失或类型错误
- **真相**：API 升级后 cassette 未同步更新
- **正确做法**：录制时记录 API 版本，定期校验 cassette 有效性

### 2.6 P02：Ollama 本地 LLM 验证（延后）

#### 方案概述
用 Ollama + Qwen2.5-7B / Llama-3.1-8B 验证提示词工程，避免生产时才发现提示词失效。

#### 延后原因

1. **模型下载需网络访问**
   - Ollama 模型从 ollama.com 下载，需国外网络访问
   - Qwen2.5-7B 模型 ~4GB，下载耗时长
   - 用户当前约束下无法下载

2. **硬件要求高**
   - 7B 模型需 8GB+ 显存或 16GB+ 内存
   - 用户硬件未知，可能无法运行

3. **验证结果仅供参考**
   - LocalLLM 与真实 Claude/GPT 输出质量差异大
   - 提示词在 LocalLLM 上通过 ≠ 在 Claude/GPT 上通过
   - 仅能验证"提示词格式正确"，不能验证"输出质量"

#### 启用时机
- 用户获得网络访问 + 16GB+ 内存后
- 作为 Phase 2.5 的可选验证手段

#### 避坑指南

**坑 1：Ollama 模型下载卡死**
- **症状**：`ollama pull qwen2.5:7b` 长时间无响应
- **真相**：oollama.com 在国内访问不稳定
- **正确做法**：用国内镜像（如 modelscope）或 VPN 下载

**坑 2：LocalLLM 验证结果误导**
- **症状**：LocalLLM 验证通过，生产环境 Claude/GPT 输出质量差
- **真相**：模型能力差异，LocalLLM 无法完全模拟 Claude/GPT
- **正确做法**：LocalLLM 仅验证"提示词格式"，输出质量需在 Phase 4 真实 API 上验证

## 3. 失败经验总结

### 3.1 共同失败模式

| 失败模式 | 涉及方案 | 根因 | 通用避坑原则 |
|---------|---------|------|------------|
| 资源占用过大 | F01 | Docker + 8 容器在 Mac 上不可行 | Mac 本地开发避免重资源容器化 |
| 性能损耗 | F01、F04 | Docker volume / 同步串行 | 选择性能更优的方案（asyncio / 原生进程） |
| 兼容性风险 | F02 | Gitea Actions vs GitHub Actions | 用 `act` 模拟，保持 workflow 一致 |
| 风险覆盖不足 | F03 | 仅 Top 3 风险 | 完整风险矩阵 + 显式技术债务 |
| 网络依赖 | P01、P02 | 首次录制 / 模型下载需网络 | 优先用人工 fixture，网络可用后再启用 |
| 验证失真 | P02 | LocalLLM ≠ 真实 LLM | 区分"格式验证"与"质量验证" |

### 3.2 通用避坑原则

1. **Mac 本地开发避免重资源容器化**
   - 优先用原生进程（Flask / Hugo server / Python script）
   - 容器化仅用于生产部署

2. **优先选择性能更优的方案**
   - asyncio > 同步
   - 原生进程 > Docker volume
   - diskcache > 重复调用

3. **用 `act` 替代 Gitea Actions**
   - 与 GitHub Actions 共享 workflow 文件
   - 行为一致，无兼容性风险

4. **完整风险防控**
   - 15 项风险矩阵（不要偷懒）
   - 8 项技术债务显式登记
   - BCP/DRP 文档化

5. **网络依赖方案延后**
   - 当前阶段用人工 fixture
   - 网络可用后切换 VCR.py / Ollama

6. **区分"格式验证"与"质量验证"**
   - LocalLLM 验证格式
   - 真实 API 验证质量

### 3.3 后续避坑检查清单

在后续开发中，遇到以下情况需检查是否踩坑：

- [ ] 是否引入了 Docker？→ 检查是否必要，能否用原生进程替代
- [ ] 是否使用 Gitea Actions？→ 切换到 `act`
- [ ] 是否仅关注 Top 3 风险？→ 补充完整风险矩阵
- [ ] 是否使用同步版管道？→ 切换到 asyncio
- [ ] 是否尝试 VCR.py 录制？→ 检查网络是否可用，否则延后
- [ ] 是否用 LocalLLM 验证提示词？→ 标注"格式验证"，质量验证留到 Phase 4

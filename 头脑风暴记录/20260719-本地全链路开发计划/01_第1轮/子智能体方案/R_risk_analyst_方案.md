# 风险分析师方案 — 第 1 轮

> 角色：风险分析师（R_risk_analyst）
> 视角：潜在风险枚举 / 影响范围评估 / 概率估算 / 风险缓解策略 / 应急预案设计
> 输出范围：本地全链路开发计划的风险矩阵与应急预案
> 编写日期：2026-07-19

---

## 角色定位

风险分析师，聚焦**风险识别与应急预案**。在本项目中负责：

1. 识别"本地全链路模拟"的全部潜在风险（按概率×影响量化）
2. 评估国外 API 不可达、Mock 失真、Hugo 构建等关键风险
3. 设计风险缓解策略与应急预案（BCP/DRP）
4. 设计本地监控告警体系（如何在本地模拟生产监控）
5. 与 07_风险管理与合规方案.md 对齐
6. 为每个开发步骤标注风险等级与应对措施

**核心立场**：风险必须量化（概率 × 影响 = 风险分），应急预案必须可执行（明确触发条件、负责人、恢复时间目标）。

## 方案清单

### 方案1：分级风险防控 + 显式技术债务登记方案（推荐 — 风险严谨派）

**核心思路**（295 字）：

建立四级风险防控体系：(1)**预防层** — 在开发计划中显式登记 8 项技术债务（TD-001~008），每项注明妥协内容、偿还时间、责任人；(2)**检测层** — 本地监控脚本 `scripts/perf_monitor.py` 采集构建时间、内存占用、Mock 响应延迟等关键指标，超标红色告警；(3)**响应层** — 每个极高风险项配置 BCP（业务连续性计划）和 DRP（灾难恢复计划），明确触发条件、负责人、恢复时间目标（RTO）；(4)**复盘层** — 每周风险评审会议，更新风险矩阵，新增风险项。与 07_风险文档对齐，确保开发阶段风险和生产风险统一管理。该方案核心优势是"风险显式可追溯"，避免隐性债务积累。

**实施步骤**：

1. **Phase A：技术债务登记表**
   - 创建 `tech-debt-ledger.md` 文件
   - 登记 8 项显式技术债务（TD-001~008）
   - 每项包含：妥协内容、偿还时间、责任人、验证方式
   - 在 Makefile 中加入 `make debt-review` 命令

2. **Phase B：本地监控体系**
   - 实现 `scripts/perf_monitor.py`：采集 CPU/内存/磁盘/构建时间
   - 实现 `scripts/health_check.sh`：检查 Mock Server/Hugo/管道健康状态
   - 配置告警阈值：内存 > 4GB、构建 > 15min、Mock 响应 > 500ms
   - 输出 `09_报告/监控看板.html`（本地 HTML，无外部依赖）

3. **Phase C：BCP/DRP 应急预案**
   - 为 R01-R04 四项极高风险编写 BCP/DRP 文档
   - 每个 BCP 包含：触发条件、应急响应步骤、负责人、RTO、回滚方案
   - 文档存储在 `docs/应急响应预案.md`

4. **Phase D：风险对齐与同步**
   - 与 07_风险管理与合规方案.md 对齐
   - 补充开发阶段特有的风险（Mock 失真、Hugo 构建等）
   - 建立风险同步机制：每月更新风险矩阵

5. **Phase E：CI 风险门禁**
   - 在 `make ci` 中加入风险检查步骤
   - 检查技术债务偿还进度
   - 检查监控指标是否达标
   - 检查 BCP/DRP 文档是否更新

**风险防控措施**：

| 风险点 | 检测方法 | 预警阈值 | 应急预案 | 恢复时间目标 |
|--------|---------|---------|---------|------------|
| Mock API 响应失真 (R01) | 字段对比脚本 | 差异率 > 5% | 切换真实 API（如可访问）或更新 Mock | < 30min |
| 用户长期无网络 (R02) | 项目进度检查 | Phase 4 阻塞 > 2 周 | 启动 VPN/代理方案或委托运维 | < 7 天 |
| Hugo 构建 OOM (R03) | psutil 内存采样 | 内存 > 4GB | 启用分批构建+--cacheDir | < 10min |
| 提示词失效 (R04) | 输出质量评分 | 中立性 < 0.7 | 切换 LocalLLM 验证或延迟到生产 | < 1 天 |
| Mock 数据不合规 (R05) | 法务评审 | 采集渠道变更 | 重新采集合规数据 | < 3 天 |
| Flask 并发崩溃 (R06) | wrk 压测 | 错误率 > 1% | 切换 gunicorn -w 4 | < 5min |
| Provider 接口错误 (R07) | 垂直切片测试 | 测试失败 | 重新设计接口+回归测试 | < 1 天 |
| asyncio fd 耗尽 (R08) | ulimit 检查 | fd > 1000 | ulimit -n 65535+Semaphore 降级 | < 5min |
| 数据合规违规 (R09) | 法务审计 | 违规项 > 0 | 立即下线内容+整改 | < 1 天 |
| API Key 泄露 (R10) | Git 历史扫描 | 检测到密钥 | 立即轮换密钥+清理历史 | < 1 小时 |

**预期效果**：

- 8 项技术债务显式登记，避免隐性积累
- 4 项极高风险有完整 BCP/DRP，RTO 明确
- 本地监控 100% 覆盖关键指标，超标自动告警
- 与 07_风险文档对齐，风险统一管理

**潜在风险**：

- **风险 1：风险矩阵维护成本高**：每周评审 + 月度更新需投入时间。**应对**：自动化风险检测脚本，减少人工评审
- **风险 2：BCP/DRP 文档过时**：应急预案未及时更新。**应对**：每季度演练一次 BCP/DRP，验证有效性
- **风险 3：监控脚本本身故障**：监控失效导致风险未被发现。**应对**：监控脚本的监控（meta-monitoring）
- **风险 4：技术债务偿还拖延**：债务积累超出可控范围。**应对**：每月债务评审，逾期债务升级为 P0

**成本估算**：

- 人力：10 人天（债务登记: 2 + 监控体系: 3 + BCP/DRP: 3 + 对齐同步: 1 + CI 门禁: 1）
- 时间：1.5 周（与开发并行）
- 技术债务：无新增（本方案就是债务管理）
- 维护：长期投入 2-3 小时/周（风险评审 + 监控维护）
- 财务：本地 $0

**致命失败场景**：

1. **场景 A**：监控脚本本身存在 bug，导致所有风险检测失效。**触发条件**：监控脚本未测试。**应对**：监控脚本必须有单元测试，且每周人工抽查一次监控数据。
2. **场景 B**：BCP/DRP 文档与实际预案脱节，紧急时刻无法执行。**触发条件**：文档未演练。**应对**：每季度演练一次，记录演练结果并更新文档。
3. **场景 C**：技术债务偿还拖延，导致 Phase 4 生产迁移时债务集中爆发。**触发条件**：债务评审流于形式。**应对**：债务偿还进度纳入项目周报，逾期债务升级处理。
4. **场景 D**：风险矩阵过时，新风险未识别。**触发条件**：风险评审停滞。**应对**：建立风险上报渠道，任何开发者发现新风险都可即时上报。

### 方案2：风险驱动测试 + 故障注入方案（混沌工程派）

**核心思路**（287 字）：

将风险防控融入测试流程，采用混沌工程思路主动注入故障：(1)**故障注入测试** — 在 Mock Server 中注入故障（500 错误、超时、限流 429、字段缺失），验证管道的容错能力；(2)**风险驱动测试用例** — 每个高风险项对应一组测试用例（如 R01 对应"Mock vs 真实 API 差异检测测试"）；(3)**游戏日演练** — 每月一次"故障演练日"，模拟极端场景（Mock Server 宕机、磁盘满、内存耗尽）。该方案核心优势是"主动发现故障"，能在本地暴露 90% 的生产问题；但故障注入用例设计复杂，需要额外投入。

**实施步骤**：

1. **Phase A：故障注入 Mock Server**
   - 扩展 Flask Mock Server，支持故障注入：
     - `?fail=500` 返回 500 错误
     - `?delay=5000` 延迟 5 秒
     - `?rate_limit=429` 返回 429 限流
     - `?missing_field=true` 返回缺失字段的 JSON
   - 编写 `tests/chaos/test_fault_injection.py`

2. **Phase B：风险驱动测试用例**
   - 为每个高风险项编写测试用例：
     - R01: `test_mock_vs_real_api_diff.py`
     - R03: `test_hugo_oom_recovery.py`
     - R06: `test_flask_concurrent_crash.py`
     - R08: `test_asyncio_fd_exhaustion.py`
   - 在 CI 中加入 `make chaos-test` 命令

3. **Phase C：游戏日演练**
   - 编写 `docs/游戏日演练手册.md`
   - 每月一次故障演练：模拟 Mock Server 宕机、磁盘满、内存耗尽
   - 记录演练结果，更新 BCP/DRP

**风险防控措施**：

| 风险点 | 故障注入方式 | 验证目标 | 通过标准 | 演练频率 |
|--------|------------|---------|---------|---------|
| Mock API 500 错误 (R01) | `?fail=500` | 管道优雅降级 | 不崩溃+记录错误 | 每次提交 |
| API 超时 (R06) | `?delay=5000` | 超时重试机制 | 重试 3 次后切换 | 每次提交 |
| API 限流 (R06) | `?rate_limit=429` | 限流退避 | 指数退避+降级 | 每次提交 |
| 字段缺失 (R01) | `?missing_field=true` | 数据校验 | pydantic 抛出明确错误 | 每次提交 |
| Hugo OOM (R03) | 限制内存 `ulimit -v 2G` | 分批构建启动 | 自动切换分批模式 | 每月演练 |
| 磁盘满 | 填充磁盘 | 错误处理+告警 | 优雅退出+告警 | 每月演练 |
| Mock Server 宕机 | kill 进程 | 健康检查+重启 | 自动重启+恢复 | 每月演练 |

**预期效果**：

- 主动发现 90% 的生产故障场景
- 故障注入测试覆盖率 100%（极高风险项）
- 每月游戏日演练，持续验证应急预案
- 管道容错能力量化（如：API 故障时 95% 的请求仍能完成）

**潜在风险**：

- **风险 1：故障注入测试影响正常开发**：故障注入可能导致测试不稳定。**应对**：故障注入测试独立于单元测试，仅在 `make chaos-test` 时运行
- **风险 2：游戏日演练耗时**：每月半天演练影响开发进度。**应对**：演练时间固定（如每月最后一个周五下午），提前规划
- **风险 3：故障注入用例设计复杂**：需要深入理解每个风险点。**应对**：从 Top 5 风险开始，逐步扩展

**成本估算**：

- 人力：15 人天（故障注入: 5 + 风险测试: 5 + 游戏日: 3 + 文档: 2）
- 时间：2 周
- 技术债务：故障注入用例维护成本
- 维护：长期投入 3-4 小时/周（含每月游戏日）
- 财务：本地 $0

**致命失败场景**：

1. **场景 A**：故障注入测试过于严格，阻塞正常开发流程。**触发条件**：故障阈值设置不合理。**应对**：分级设置阈值，开发模式宽松，CI 模式严格。
2. **场景 B**：游戏日演练发现严重问题但无法及时修复。**触发条件**：演练暴露的 bug 修复成本高。**应对**：演练后立即评估，严重问题升级为 P0。
3. **场景 C**：故障注入用例与真实故障不匹配。**触发条件**：真实故障模式未覆盖。**应对**：每次生产事故后补充故障注入用例。

### 方案3：最小化风险防控 + 快速迭代方案（敏捷派）

**核心思路**（265 字）：

放弃完整的风险矩阵和 BCP/DRP 文档，采用最小化风险防控：(1)**Top 3 风险聚焦** — 仅关注 R01（Mock 失真）、R02（网络不可达）、R03（Hugo OOM）三项极高风险；(2)**轻量监控** — 仅监控构建时间、内存占用、Mock 响应时间三项核心指标；(3)**口头应急预案** — 不编写完整 BCP/DRP 文档，仅在每周站会讨论风险应对；(4)**延迟风险登记** — 技术债务在遇到时再登记，不预先规划。该方案核心优势是"轻量高效"，适合快速迭代；但风险覆盖不足，可能遗漏关键风险。

**实施步骤**：

1. **Phase A：Top 3 风险应对**
   - 为 R01/R02/R03 编写简短应对方案（每个 < 1 页）
   - 在开发计划中标注关键风险点

2. **Phase B：轻量监控**
   - 在 `make test` 中加入构建时间检查（< 15min）
   - 在 `make build` 中加入内存检查（< 4GB）
   - 超标时打印警告

3. **Phase C：站会风险讨论**
   - 每周站会增加"风险讨论"环节
   - 记录到项目周报

**风险防控措施**：

| 风险点 | 应对方式 | 负责人 | 触发条件 |
|--------|---------|--------|---------|
| Mock 失真 (R01) | 生产迁移时用真实 API 验证 | 技术实现者 | Phase 4 |
| 网络不可达 (R02) | 提前规划 VPN | 项目经理 | Phase 4 前 |
| Hugo OOM (R03) | 分批构建 | 性能工程师 | 内存 > 4GB |

**预期效果**：

- 开发效率最高（无风险文档维护负担）
- Top 3 风险有基本应对
- 适合快速原型阶段

**潜在风险**：

- **风险 1：遗漏关键风险**：未识别的风险可能造成生产事故。**应对**：每月补做一次完整风险评估
- **风险 2：应急预案不充分**：紧急时刻缺乏指导。**应对**：遇到事故时临时编写
- **风险 3：技术债务积累**：未显式登记，可能失控。**应对**：每月技术债务评审

**成本估算**：

- 人力：3 人天（Top 3 应对: 1 + 监控: 1 + 站会: 1）
- 时间：0.5 周
- 技术债务：隐性积累风险高
- 维护：长期投入 0.5 小时/周
- 财务：本地 $0

**致命失败场景**：

1. **场景 A**：未识别的风险造成生产事故，无应急预案。**触发条件**：风险评估不充分。**应对**：生产迁移前补做完整风险评估。
2. **场景 B**：技术债务积累到不可控。**触发条件**：债务未登记。**应对**：每月强制债务评审。

## 方案对比与风险分析师推荐

### 三方案风险防控对比矩阵

| 风险维度 | 方案1（分级防控+债务登记） | 方案2（风险驱动+故障注入） | 方案3（最小化+快速迭代） |
|---------|------------------------|------------------------|----------------------|
| **风险覆盖度** | ⭐⭐⭐⭐⭐ 高（15 项） | ⭐⭐⭐⭐⭐ 高（含故障注入） | ⭐⭐ 低（仅 3 项） |
| **预防能力** | ⭐⭐⭐⭐ 高（显式登记） | ⭐⭐⭐⭐⭐ 最高（主动发现） | ⭐⭐ 低（被动应对） |
| **应急能力** | ⭐⭐⭐⭐⭐ 高（BCP/DRP） | ⭐⭐⭐⭐ 高（演练验证） | ⭐ 低（口头） |
| **维护成本** | ⭐⭐⭐ 中（2-3h/周） | ⭐⭐ 高（3-4h/周） | ⭐⭐⭐⭐⭐ 最低（0.5h/周） |
| **首次部署成本** | ⭐⭐⭐ 中（1.5 周） | ⭐⭐ 低（2 周） | ⭐⭐⭐⭐⭐ 最低（0.5 周） |
| **与 07 文档对齐** | ⭐⭐⭐⭐⭐ 完美对齐 | ⭐⭐⭐⭐ 较好 | ⭐⭐ 弱 |

### 风险分析师推荐

**首选方案：方案1（分级风险防控 + 显式技术债务登记方案）**

**推荐理由（压倒性优势）**：

1. **风险覆盖最完整**：15 项风险全部分级管理，与 07_风险文档对齐
2. **技术债务显式管理**：8 项 TD 登记，避免隐性积累
3. **BCP/DRP 完整**：4 项极高风险有明确应急预案
4. **维护成本可控**：2-3 小时/周，不影响开发效率
5. **与开发计划完美配合**：风险登记嵌入开发流程

**次选方案：方案2（风险驱动测试 + 故障注入）**

可作为方案1 的补充，用于主动发现故障。建议在方案1 完成后，对 Top 5 风险实施故障注入测试。

**不推荐方案3（最小化风险防控）**：

虽然开发效率最高，但风险覆盖不足，不符合"系统稳定高于一切"的核心原则。仅适合纯原型阶段，不适合有生产迁移计划的项目。

### 风险分析师最终建议

采用**方案1 为主框架**，叠加方案2 的故障注入测试：

- **方案1 提供**：完整风险矩阵 + 技术债务登记 + BCP/DRP + 监控体系
- **方案2 补充**：对 R01/R03/R06/R08 四项高风险实施故障注入测试
- **方案3 不采用**：风险覆盖不足

这样既保证了风险覆盖的完整性，又获得了主动发现故障的能力，且维护成本可控。

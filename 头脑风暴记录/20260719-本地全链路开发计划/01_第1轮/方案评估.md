## 1. 评估方法说明

### 1.1 评分标准

| 评估维度 | 权重 | 评分范围 | 评分依据 |
|---------|------|---------|---------|
| 现实可行性 | 30% | 1-10 | 当前条件下（无国外网络、Mac 本地）能否实际落地 |
| 实施简易度 | 25% | 1-10 | 分数越高越容易实施（正向），含工具链成熟度、文档质量 |
| 成本效益 | 20% | 1-10 | 4 子项平均：人力成本 / 时间成本 / 技术债务 / 维护成本（越高越好） |
| 目标契合度 | 20% | 1-10 | 与"本地全链路走通 + 看到网站内容"目标的匹配程度 |
| 创新性 | 5% | 1-10 | 方案的创新突破程度，对执行型任务为参考性质 |

### 1.2 置信度定义

- **高**：有充分证据 / 经验支撑
- **中**：有部分证据 / 逻辑推理
- **低**：主要基于假设 / 推测

置信度加权：高 ×1.0 / 中 ×0.8 / 低 ×0.6，释放权重按比例分配给高置信度维度。

### 1.3 自动去除机制

- **绝对下限**（满足任一即去除）：可行性 ≤2 / 简易度 ≤2 / 成本效益 ≤2 / 契合度 ≤3
- **相对阈值**：加权总分低于本轮中位数 ×0.6 标记"待去除"
- **创新保护**：创新性 ≥7 且可行性 =3 时，给予一轮进化机会

## 2. 方案详细评估

### 方案 1：本地全链路开发主框架（推荐主方案）

| 评估维度 | 评分(1-10) | 评估说明 | 置信度 |
|---------|-----------|---------|--------|
| 现实可行性 | 9 | 完全不依赖国外网络，Flask Mock + Provider 抽象是成熟模式，pydantic-settings 双轨制在 Python 生态中验证充分 | 高 |
| 实施简易度 | 8 | 技术栈（Hugo/Flask/pytest）文档完善，工具链成熟；唯一复杂点是 asyncio 重构和 Hugo 分批构建 | 高 |
| 成本效益 | 8 | 人力 45 人天合理；时间 5 周适中；技术债务显式登记 8 项可追溯；维护长期投入低（双轨制切换零代码） | 高 |
| 目标契合度 | 10 | 完全匹配"本地全链路走通 + 看到网站内容"目标；Phase 0-3 都在本地完成，Phase 4 才需网络 | 高 |
| 创新性 | 6 | Provider 抽象 + 配置双轨制是经典模式，无重大创新，但 22 项性能预算嵌入每一步是创新点 | 中 |
| **加权总分** | **8.55/10** | 可行性 2.7 + 简易度 2.0 + 成本 1.6 + 契合度 2.0 + 创新 0.3 = 8.6（创新按 0.8 折算后 8.55） | |

**评估结论**：保留（主方案）

**主要支持角色**：全部 5 角色（技术实现者 / 测试架构师 / 战略架构师 / 风险分析师 / 性能工程师）

**核心优势**：
- 33 项编码任务清单，每项含文件路径 / 函数签名 / 验收命令，可直接执行
- 5 阶段划分清晰，入站 / 出站条件明确
- 11 项本地→生产迁移路径，回滚方案完整
- 测试金字塔（单元 70% + 集成 20% + E2E 10%）+ 6 份 OpenAPI 契约
- 15 项风险矩阵 + 8 项技术债务显式登记

**潜在不足**：
- asyncio 重构对开发者要求较高（需熟悉异步编程）
- Hugo 10 万页构建在 8GB Mac 上仍有 OOM 风险
- 450 个 Mock fixture 文件维护成本不小

### 方案 2：AI 提示词验证增强包（Ollama + 故障注入）

| 评估维度 | 评分(1-10) | 评估说明 | 置信度 |
|---------|-----------|---------|--------|
| 现实可行性 | 6 | Ollama 本地 LLM 需 16GB+ 内存，用户硬件未知；故障注入测试设计复杂 | 中 |
| 实施简易度 | 5 | Ollama 模型下载（需网络）+ 调优；故障注入用例需深入理解每个风险点 | 中 |
| 成本效益 | 6 | 人力 +7 人天合理；但模型下载需网络可能与"无国外网络"约束冲突 | 中 |
| 目标契合度 | 7 | 提示词验证是真实需求，但非"走通全链路"的必要条件；故障注入属增强项 | 高 |
| 创新性 | 8 | Ollama 模拟生产 LLM + 故障注入测试是较创新的本地验证手段 | 中 |
| **加权总分** | **6.20/10** | 可行性 1.8 + 简易度 1.25 + 成本 1.2 + 契合度 1.4 + 创新 0.4 = 6.05（置信度折算后 6.20） | |

**评估结论**：保留（可选增强包，按需启用）

**启用条件**：
- 用户本地 16GB+ 内存
- 已通过 VPN 或代理获得 Ollama 模型下载能力
- 对提示词工程有强烈验证需求

**主要支持角色**：技术实现者 + 风险分析师

**核心价值**：
- 在本地验证提示词工程，避免生产时才发现提示词失效
- 故障注入测试主动发现 90% 的生产故障场景

**潜在风险**：
- Ollama 模型下载需网络访问，可能与用户约束冲突
- 故障注入用例设计复杂，需 5 人天投入
- LocalLLM 与真实 Claude/GPT 输出质量差异，验证结果仅供参考

### 方案 5：Docker Compose 环境对等方案（不推荐）

| 评估维度 | 评分(1-10) | 评估说明 | 置信度 |
|---------|-----------|---------|--------|
| 现实可行性 | 4 | Docker Desktop 在 Mac 上资源占用大；volume 性能损耗导致 Hugo 构建慢 2-3 倍 | 高 |
| 实施简易度 | 4 | 8 个容器编排复杂；Gitea Actions 与 GitHub Actions 兼容性风险 | 高 |
| 成本效益 | 4 | 人力成本高；维护成本大；性能损耗明显 | 高 |
| 目标契合度 | 5 | 环境对等是好事，但与"本地走通"目标相比过度工程化 | 高 |
| 创新性 | 5 | 容器化是常规方案，无创新 | 低 |
| **加权总分** | **4.35/10** | 可行性 1.2 + 简易度 1.0 + 成本 0.8 + 契合度 1.0 + 创新 0.25 = 4.25（置信度折算后 4.35） | |

**评估结论**：去除（不推荐）

**去除理由**：
1. Docker Desktop 在 Mac 上资源占用大（建议 16GB+），用户硬件未知
2. volume 性能损耗导致 Hugo 构建慢 2-3 倍，与性能预算冲突
3. Gitea Actions 与 GitHub Actions 兼容性风险，可能引入隐性债务
4. 容器编排复杂度高，维护成本大
5. 与"最小化原则"和"风险最小化"原则冲突

**避坑指南**：
- 不要为了"环境一致性"而引入 Docker，本地开发用 Flask Mock + Hugo server 足够
- 如确实需要容器化，建议仅在生产部署阶段使用 Docker（如 Cloudflare Pages 不支持时）
- 不要使用 Gitea Actions 替代 GitHub Actions，兼容性问题不值得

## 4. 评估结论

### 4.1 推荐方案组合

**主方案 + 增强包 3 + 增强包 4**（即"主方案 + 自动化测试增强 + 垂直切片"）

- 加权总分：8.55 + 7.80 + 7.70 = 24.05（三方案平均 8.02）
- 总人力：45 + 6 + 0（垂直切片作为 Phase 2.5 补充，增量成本已含在主方案）= 51 人天
- 总时间：5.5 周
- 财务：$0（本地）

### 4.2 关键决策点

| 决策点 | 建议 | 决策时机 |
|--------|------|---------|
| 是否启用方案 2（Ollama） | 视本地内存而定，16GB+ 可启用 | Phase 0 评估后 |
| 是否启用方案 3（自动化测试） | 强烈建议启用 | Phase 1 开始时 |
| 是否启用方案 4（垂直切片） | 强烈建议启用 | Phase 2.5 |
| 方案 5（Docker） | 不采用 | — |
| 方案 6（VCR.py） | 延后到 Phase 4 | Phase 4 |

### 4.3 风险传染检测回顾

5 项共享假设已在方案汇总中标注建议，本评估确认：

1. **Mock 数据能代表真实 API** → Phase 4 用 VCR.py 验证
2. **Hugo 能在本地构建 10 万页** → Phase 0 评估机器内存，必要时降级到分批构建
3. **Provider 接口设计正确** → Phase 2.5 用垂直切片验证
4. **用户能获得国外网络访问** → 提前与用户确认 VPN/委托运维方案
5. **Mac fd 上限足够** → `ulimit -n 65535` 提前设置

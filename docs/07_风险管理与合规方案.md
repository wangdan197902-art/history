#### 2.2.2 数据丢失

**Git 仓库保护：**
- GitHub 主仓库（实时同步）
- 本地 clone（每日 pull）
- Cloudflare R2 备份（每日 upload）

**关键数据：**
- `content/` — Markdown 内容（Git 版本化）
- `data/events/*.json` — 事件数据（Git 版本化）
- `data/countries.json` — 地区元数据（Git 版本化）
- `prompts/` — Prompt 模板（Git 版本化）
- 邮件订阅列表 — Buttondown 后台（API 导出）

**恢复流程：**
1. 从 GitHub 恢复最新 commit
2. 从 R2 恢复当日数据（如需）
3. 重新部署到 Cloudflare Pages
4. 验证站点完整性

#### 2.2.3 域名安全

**保护措施：**
- Namecheap 账号 2FA
- 域名锁定（Domain Lock）
- WhoisGuard 隐私保护
- DNS 转入 Cloudflare（DNS 锁定）
- 注册 5+ 年（避免忘记续费）

### 3.2 战争 / 冲突事件地区视角差异处理

**核心原则：** 战争事件必须呈现多方视角，避免单一立场。

**处理 SOP：**

1. **AI 识别**：关键词检测（war, battle, invasion, occupation 等）→ 标记 `sensitivity_flags: ["war"]`
2. **多源采集**：至少引用 2 个不同国家的来源
   - 例：俄乌冲突 → 俄罗斯 Wikipedia + 乌克兰 Wikipedia + 国际媒体
3. **Claude 重写**：Prompt 强制要求"present multiple perspectives"
4. **中立性评分**：< 0.7 必须人工复审
5. **人工审核**：编辑 + 历史顾问双签
6. **术语规范**：
   - 使用"军事冲突"而非"侵略战争"（除非有联合国决议）
   - 使用"武装力量"而非"恐怖分子"（除非有国际法院判决）
   - 使用"争议地区"而非"被占领土"（除非引用具体来源）

**典型案例：**

| 事件 | 处理方式 |
|------|---------|
| 第二次世界大战 | 各国视角（美/英/苏/德/日/中） |
| 俄乌冲突 | 俄方 + 乌方 + 国际社会视角 |
| 朝鲜战争 | 朝鲜 + 韩国 + 美国 + 中国视角 |
| 中东冲突 | 以色列 + 巴勒斯坦 + 国际视角 |
| 福克兰战争 | 阿根廷 + 英国视角 |

### 3.3 政治事件中立性原则

**核心原则：** 政治事件采用学术化、事实化表述，不评判对错。

**处理规则：**

| 政治事件类型 | 表述规范 | 禁用词汇 |
|------------|---------|---------|
| 革命 | "X 国发生革命，导致 Y 政府下台" | "正义革命" / "反革命暴乱" |
| 选举 | "X 当选总统，得票率 Y%" | "当选救星" / "舞弊当选" |
| 政变 | "军方发动政变，推翻 X 政府" | "解放政变" / "暴政政变" |
| 抗议 | "X 地发生抗议活动，诉求为 Y" | "暴徒骚乱" / "民主运动" |
| 政府更迭 | "X 党赢得选举，组建新政府" | "伟大胜利" / "黑暗时代" |

### 3.4 宗教事件尊重原则

**核心原则：** 宗教事件客观描述，不评判教义对错。

**处理规则：**

- 使用"宗教信徒认为..."而非"事实上..."
- 引用宗教经典时标注"据《XX》记载"
- 宗教冲突描述双方立场
- 避免亵渎性表述
- 宗教人物使用尊称（如"先知穆罕默德"、"耶稣基督"）

### 3.5 争议领土 / 事件表述规范

**核心原则：** 使用国际法认可的术语，引用联合国文件。

**领土争议术语表：**

### 3.7 AI 改写抄袭防范

**风险：** Claude 改写可能保留 Wikipedia 原文结构，被识别为抄袭。

**防范措施：**

1. **Prompt 强制改写**：
   - "Rewrite in your own words, NOT Wikipedia verbatim"
   - "Use different sentence structure"
   - "Add narrative context"

2. **原创性检测**：
   - 工具：Copyscape / Grammarly
   - 频率：每月 1% 抽检
   - 阈值：相似度 < 30%

3. **多源融合**：
   - 不只引用 Wikipedia，加入国家档案馆、BBC、NYT 等
   - Claude 综合多源改写

4. **引用规范**：
   - 每个事件标注来源
   - 直接引用加引号 + 来源
   - CC BY-SA 4.0 许可证声明

### 4.2 GDPR / CCPA 合规

#### 4.2.1 GDPR（欧盟通用数据保护条例）

**适用范围：** 任何处理欧盟用户数据的网站

**核心要求：**

1. **合法依据**：邮件订阅需用户明确同意（consent）
2. **数据最小化**：仅收集必要数据（邮箱）
3. **透明性**：隐私政策明确告知数据用途
4. **用户权利**：
   - 访问权（获取个人数据）
   - 更正权（修改错误数据）
   - 删除权（被遗忘权）
   - 数据可携权（导出数据）
   - 反对权（停止处理）
5. **数据保护官（DPO）**：如需要，指定 DPO
6. **数据泄露通知**：72 小时内通知监管机构

**实施措施：**

- **邮件订阅双重 opt-in**：
  1. 用户填写邮箱 → 发送确认邮件
  2. 用户点击确认链接 → 订阅成功
  3. 记录同意时间 + IP + 来源

- **隐私政策页面** `/privacy/`：
  - 数据控制者：[公司名 / 个人]
  - 数据类型：邮箱（订阅）、Cookie（分析）
  - 用途：发送订阅邮件、流量分析
  - 第三方共享：Buttondown、Google Analytics
  - 用户权利：访问 / 更正 / 删除 / 反对
  - 联系方式：privacy@today-in-history.example.com

- **Cookie consent banner**：
  - 仅必要 cookie 默认开启
  - 分析 cookie 需用户同意
  - 提供"全部接受" / "仅必要" / "管理偏好"选项

- **数据删除请求 API**：
  ```python
  # 用户请求删除数据
  POST /api/v1/privacy/delete-request
  {
    "email": "user@example.com",
    "reason": " GDPR Article 17 right to erasure"
  }
  # 72 小时内从 Buttondown + GA4 + 备份中删除
  ```

#### 4.2.2 CCPA（加州消费者隐私法）

**适用范围：** 处理加州居民数据的网站

**核心要求：**

1. **知情权**：告知收集哪些数据
2. **删除权**：用户可要求删除
3. **退出销售权**：用户可要求不销售其数据
4. **非歧视权**：行使权利不受歧视

**实施措施：**

- "Do Not Sell My Personal Information" 链接（页脚）
- 隐私政策披露数据收集范围
- 用户数据查询 / 删除请求 API

#### 4.3.2 AdSense 风险场景

**风险 1：AI 内容被识别为低质**

- 应对：严格审核 + 敏感事件关键词过滤

**风险 3：点击欺诈**
- 应对：不点击自己广告 + 监控异常点击

**风险 4：违规流量来源**
- 应对：不购买流量 + 不交换点击

### 4.4 FTC 披露合规

#### 4.4.1 FTC 披露要求

**适用范围：** 美国联邦贸易委员会要求的所有付费推广

**核心要求：**

1. **明确披露**：付费链接必须明确标注
2. **显著位置**：披露在链接附近，用户易见
3. **通俗易懂**：使用"#ad" / "Sponsored" / "Affiliate Link"
4. **视频/音频**：开头 + 描述双披露

**实施措施：**

- **Amazon 联盟披露**（页脚 + 每个联盟链接附近）：
  > "As an Amazon Associate, I earn from qualifying purchases."

- **赞助内容披露**：
  > "Sponsored: This content is sponsored by [Sponsor Name]"

- **AI 内容披露**：
  > "This content was AI-assisted and reviewed by human editors."

### 4.5 公共领域内容使用规范

#### 4.5.1 公共领域判定

**Wikipedia 文本：**
- 许可证：CC BY-SA 4.0
- 使用规则：署名 + 相同许可 + 标注修改
- 实施：AI 改写为原创叙事 + 标注 Wikipedia 来源

**Wikimedia Commons 图片：**
- 许可证类型：
  - Public domain（自由使用）
  - CC BY 4.0（署名）
  - CC BY-SA 4.0（署名 + 相同许可）
  - CC BY-NC 4.0（署名 + 非商业）❌ 不可用
  - CC BY-ND 4.0（署名 + 不修改）❌ 不可用

**判定流程：**
1. 检查图片元数据 `LicenseShortName`
2. 仅接受：Public domain / CC BY 4.0 / CC BY-SA 4.0
3. 拒绝：CC BY-NC / CC BY-ND / Fair use

#### 4.5.2 历史事实无版权

**原则：** 历史事实本身无版权，任何人可自由使用。

**实施：**
- 事件年份、地点、人物：无版权
- 事件描述：AI 改写为原创
- 直接引用：加引号 + 来源

### 5.2 商业风险应对

#### 5.2.1 收入不达标

**触发条件：** 连续 2 个月收入低于预期 50%

**应对措施：**

1. **诊断**：分析哪个变现渠道出问题
   - 广告：RPM 下降 / 流量下降 / 广告屏蔽
   - Amazon：转化率下降 / 推荐位置不佳
   - 邮件：打开率下降 / 退订率高
   - API：用户少 / 定价高

2. **优化**：
   - 流量：SEO 优化 + Reddit / HN 推广
   - 转化：A/B 测试广告位 / CTA
   - 多元化：增加邮件赞助 / PDF 年鉴

3. **降本**：
   - AI 成本优化
   - 减少外包
   - 暂停扩展

#### 5.2.2 AI 成本超预算

**触发条件：** 月度 AI 成本 > $100（Phase 2-3）

**应对措施：**

1. **缓存优化**：30 天缓存 + 增量更新
2. **降级模型**：
   - 非核心内容用 Claude Haiku
   - 翻译用 GPT-4o-mini
3. **减少语种**：从 10 减到 5
4. **减少地区**：从 30 减到 15
5. **批量调用**：合并多个事件为单次 API

### 13.3 敏感事件处理 SOP 模板

```markdown
# Sensitive Event Handling SOP

## 1. Identification

### 1.1 Automatic Detection
The audit system flags events containing:
- War/military keywords
- Political repression keywords
- Religious conflict keywords
- Territorial dispute keywords
- Terrorism keywords

### 1.2 Manual Flagging
Editors can manually flag any event for sensitivity review.

## 2. Classification

### L1: Extreme Sensitivity
- Holocaust / genocide / massacre
- Mandatory: Human review + historical advisor + legal advisor

### L2: High Sensitivity
- Political repression / war crimes
- Mandatory: Human review + historical advisor

### L3: Medium Sensitivity
- War / coup / revolution
- Standard: AI review + 5% sampling

### L4: Low Sensitivity
- General political events
- Standard: AI review + 1% sampling

## 3. Review Process

### Step 1: AI Audit
- Fact check (≥2 sources)
- Neutrality score (≥0.7 to pass)
- Sensitivity flag detection

### Step 2: Human Review (L1/L2)
- Editor reviews content
- Historical advisor verifies accuracy
- Legal advisor reviews compliance

### Step 3: Revision
- If neutrality < 0.8: Revise with Claude
- Re-audit after revision

### Step 4: Publication
- Publish with sensitivity flag
- Monitor user feedback

## 4. Emergency Takedown

If a sensitive event is incorrectly published:
1. Immediately remove (git rm + redeploy)
2. Replace with "Content under review" placeholder
3. Investigate root cause
4. Revise Prompt if needed
5. Republish when correct
6. Document in error case library

## 5. Country-Specific Procedures

### Germany (de)
- Nazi content: Use "National Socialist regime"
- No display of Nazi symbols (even historical photos must be censored)
- Reference Holocaust Memorial Museum

### China (cn)
- Tiananmen: Multiple sources, neutral tone
- Cultural Revolution: Academic references
- Taiwan: "Taiwan (disputed political status)"

### Turkey (tr)
- Armenian events: "1915 events" / "Armenian deportations"
- Cite international scholarship

### Russia (ru) / Ukraine (ua)
- Use neutral terminology
- Cite UN and international law
- Present both perspectives
```

Today in History Archive uses Artificial Intelligence (AI) to assist in content creation. This disclosure explains how AI is used and our human oversight process.

1. **Claude 3.5 Sonnet** (Anthropic)

   - Purpose: Translation and content auditing

3. **Gemini 1.5 Flash** (Google)
   - Purpose: Historical scene illustration (when Wikimedia images unavailable)
   - Used for: AI-generated illustrations, clearly labeled

## Human Oversight

All AI-generated content is reviewed by human editors:

1. **Daily Review**: 5% of major country content is sampled
2. **Sensitive Events**: 100% of flagged events are reviewed
3. **User Feedback**: All user-reported errors are investigated within 48 hours
4. **Monthly Audit**: Translation quality is sampled by native speakers

## Content Quality Standards

- **Neutrality Score**: AI self-evaluation, must be ≥0.7 (0-1 scale)
- **Source Verification**: At least 2 independent sources per event
- **Original Writing**: AI rewrites content in its own words, not Wikipedia verbatim
- **Fact Check**: Cross-verification with multiple sources

AI may make errors. We acknowledge:
- Factual inaccuracies are possible
- Translation quality varies by language
- Sensitive topic handling requires human judgment
- AI-generated images may contain inaccuracies

- Subject: "AI Content Error Report"
- Include: Page URL, error description, suggested correction

We commit to:
- Acknowledge within 24 hours

- Document in our error case library

## Updates
